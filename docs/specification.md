# UNCHI GF: Absolute Sync Edition 基本仕様書

## 1. プロジェクト概要
`unchi-gf` は、トイレをテーマにしたカジュアルなカードバトルゲームです。オンライン対戦、CPU戦、ローカル対戦をサポートし、シンプルながらも「プレミアムな便所体験」を視覚的に提供することを目的としています。

## 2. 技術スタック
プロジェクトは保守性を高めるため、以下の技術をCDN経由で使用し、単一のHTMLファイルで完結する構成を基本としています。

- **Core**: React 18 (UMD)
- **Logic**: Babel Standalone (JSXトランスパイル)
- **Styling**: Tailwind CSS
- **Backend/Realtime**: Supabase (supabase-js)

## 3. デザインシステム

### 3.1 カラーパレット
一貫した「トイレ感」と「視認性」を維持するための指定色です。

| 要素 | 色 / クラス | HEX/RGBA (参考) | 役割 |
| :--- | :--- | :--- | :--- |
| **メイン背景** | `.bg-toilet` | `#4e342e` | 便所の壁をイメージした茶色 |
| **背景パターン** | `radial-gradient` | `#3e2723` | タイルまたはドット模様 |
| **チームA (赤)** | `.team-a`, `.badge-a` | `#ef4444` | 1P側またはチームAの識別色 |
| **チームB (青)** | `.team-b`, `.badge-b` | `#3b82f6` | 2P側またはチームBの識別色 |
| **ハイライト** | `text-yellow-500` | `#eab308` | タイトル、重要なテキスト、勝者 |
| **MPバー** | `.bar-mp` | `#00b0ff` | マナ（便意）の表示 |
| **カード背景** | `.bg-stone-100` | `#f5f5f4` | 基本カードの清潔感ある白 |
| **レアカード背景** | `.bg-yellow-100` | `#fef9c3` | 特殊スキルの背景 |

### 3.2 タイポグラフィ
- **フォント**: `font-sans` をベースに、ほぼ全ての要素で `font-black` (最太) を使用します。
- **スタイル**: `uppercase`, `italic`, `tracking-widest` を組み合わせ、インパクトのあるデザインにします。

### 3.3 コンポーネント定義
- **プレイヤーカード**: `rounded-2xl`, `border-2`, `p-card-bg` を使用。アクティブ時は `p-card-active` (黄色発光) を適用。
- **スキルボタン**: `card-shadow` (下に4pxの影) を適用し、クリック時に `translate-y-1` または影を消すことで「押し込み感」を演出。
- **ログエリア**: `bg-[#3e2723]`, `border-l-4 border-yellow-600` で強調。

## 4. UI/UX ルール

### 4.1 アニメーション
ユーザーフィードバックを視覚的に伝えるため、以下のクラスを維持します。
- `animate-float-up`: ダメージや回復値が上に消えていく演出。
- `shake`: 攻撃を受けた際などの揺れ演出。
- `animate-unchi-spin`: ロード中やリフレッシュ時の回転演出。

### 4.2 フィードバック
- **非活性状態**: MP不足や自分のターンでない場合は `opacity-20` および `grayscale` を適用し、選択不可であることを明示します。
- **インタラクション**: ボタンには必ず `active:scale-95` 等の微小なスケール変化を追加し、触感を与えます。

### 4.3 レイアウト
- **コンテナ**: `max-width: 448px` (max-w-md) を基準とし、モバイル端末でのプレイを最適化します。
- **スクロール**: 画面全体は `overflow-hidden` とし、カードリストやログなどの特定エリアのみスクロール可能にします。

## 5. ロジックとデータ整合性

### 5.1 ステート管理
- `gameState` オブジェクトに `version` プロパティを持たせ、オンライン同期時の競合を防ぎます。
- ステート更新はホスト（`isHost: true`）が統括し、ブロードキャスト経由でクライアントに同期します。

### 5.2 バトルフロー
1. スキル選択
2. ターゲット選択（必要な場合）
3. `processBattle` 実行
4. ステート更新（HP/MP計算、ターン移行、ログ生成）
5. 同期（Supabase経由）

## 6. 今後の拡張ガイドライン
- **新機能追加時**: 既存の `specification.md` に定義されたカラーパレットとコンポーネントクラスを再利用してください。
- **CSSの変更**: `index.html` 内の `<style>` タグに集約し、Tailwindのユーティリティで解決できない複雑なアニメーションのみ記述してください。
- **モジュール化**: ファイルサイズが肥大化した場合は、コンポーネントごとに分割を検討しますが、単一ファイルでの動作（ポータビリティ）を優先してください。
